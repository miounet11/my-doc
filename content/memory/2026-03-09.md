## Heartbeat: 主控成长协同计划

- 汇总到的最新状态显示：当前系统的核心矛盾仍然不是功能缺口，而是监督、证据、记忆分级和主脑能力回收还不够硬。
- main 长期记忆与昨日记录一致指向四个共同短板：
  1. 假成功/无效成功信号仍会污染判断
  2. P0 记忆门槛不够严格
  3. agent / skill / subagent / session 的决策边界不统一
  4. Codex / Claude 仍容易替代主脑的规划与判断
- 现有可见 session 很少，说明当前并没有一个活跃、可观测、分工明确的 agent 群在稳定并行推进；更接近“主脑已定义问题，但协作面板还没真正立起来”。
- 因此下一轮最合理的协同方向，不是盲目增派 agent，而是先把主控层的分工协议和证据板做实。

### 下一轮优先级

#### P0
- reviewer：优先负责“假成功清除”与 evidence-first 检查表，任何成功结论必须挂证据。
- goal-manager：输出统一的 agent 决策金字塔，明确 skill / tools / main / subagent / ACP session 的使用边界。
- world-model：梳理 `projects/原始` 当前真实状态面板，形成 evidence board 所需的最小字段集合（进程、日志、测试、页面/服务、最近失败）。
- main：收紧 MEMORY.md 的 P0 条目，只保留长期稳定、验证充分、错误代价高的内容。

#### P1
- coder / implementer：在 reviewer 和 world-model 给出约束后，再落地 evidence board 模板与检查脚本。
- tester：补一层“学习效果回测”最小机制，验证某项修复是否真的减少同类错误。

#### P2
- evolution：等 P0/P1 跑顺后，再总结哪些能力真正从外部执行者回收到 OpenClaw 主脑。

### 分工原则
- 主脑负责：优先级、标准、收口、长期记忆。
- reviewer 负责：拆穿假进展。
- world-model 负责：系统现实状态与风险地图。
- goal-manager 负责：下一步排序与 agent 边界。
- coder / Claude / Codex 只负责实现，不负责替代主脑做最终判断。

### 结论
- 下一轮不追求“更多任务同时开”，而追求“更少但更真”。
- 只要 P0 四件事落地，后续多 agent 协作才不会继续建立在脆弱叙事上。
