## 值得学的

### 1. 用“循环深度”代替一部分参数堆叠
这个 repo 最值得注意的点，不是它是不是 Mythos，而是它抓住了一个很重要的方向：

**把推理深度放到 inference time，而不是全靠更深更大的静态网络。**

也就是：
- 不是永远固定 96 层、128 层
- 而是保留一个可重复运行的 block
- 简单问题少循环，复杂问题多循环

这背后的学习点是：
- **计算预算动态分配**
- **test-time scaling**
- **参数量和推理深度解耦**

这在很多系统设计里都很有启发，不只是 LLM。

---

### 2. “隐式思维”这件事怎么工程化
它在讲一个很核心的思路：

> 不把所有中间推理都显式吐成 token，而是在 latent space 里做多步更新。

这个方向值得学，因为它对应了几个现实问题：
- 显式 CoT 很贵
- 显式 CoT 会暴露噪声路径
- token 级 reasoning 有时太离散、太早承诺

所以如果你关心更强推理模型，值得学习的是：
- 如何把“多步思考”变成**状态迭代**
- 如何让模型在内部反复 refine hidden state
- 如何让推理深度可控，而不是固定一次前向

即使这个 repo 未必代表真实 Mythos，它提出的问题是对的。

---

### 3. 把“稳定性”当成一等公民
README 里反复强调 recurrent / looped model 的训练不稳定，比如：
- residual explosion
- spectral radius 爆掉
- recurrence drift

这点很值得学，因为很多人看新架构只看“idea 酷不酷”，不看“能不能训稳”。

这里真正值得学的是一种工程习惯：

**新架构 = 表达能力 + 稳定性约束 + 可验证训练行为**

也就是别只问：
- 它能不能更聪明

还要问：
- 它会不会炸
- 哪个矩阵/路径会导致不稳定
- 有没有 by-construction 的稳定性约束
- 推理增加 loop 时会不会 overthink

这个视角很对。

### 4. 把 MoE 和 recurrence 组合起来的思路

如果不同 loop 真的走不同 expert 路由，那就意味着：
- 早期 loop 做 pattern extraction
- 中期 loop 做组合推理
- 后期 loop 做 refinement / verification

这是一种很有价值的建模想法。

## RWKV 最值得肯定的地方

### 1. 推理复杂度很漂亮
Transformer 的注意力在长上下文下很贵，KV cache 也会越来越大。

RWKV 的核心吸引力是：
- 训练时可以像 Transformer 一样并行
- 推理时像 RNN 一样维护状态
- 理论上每生成一个 token 的成本不随上下文长度线性增长

这点很重要。

如果你想做：
- 超长上下文
- 端侧模型
- 长时间运行的 agent
- 低显存推理
- 流式处理

RWKV 的结构天然有吸引力。

---

### 2. 它是真正工程化过的非 Transformer 路线
很多“Transformer 替代架构”停留在论文或小模型实验。

RWKV 不一样：
- 有社区
- 有训练实践
- 有模型权重
- 有推理实现
- 有多轮版本演进
- 真的被人拿来用

这一点非常难得。

都是在试图回答：

> 能不能不用全注意力，也获得强序列建模能力？

RWKV 是其中最接近 RNN 直觉、也比较 hacker-friendly 的路线。

---

### 3. 它对“持续状态”很友好
Transformer 本质上更像“每次带着一大摞上下文重算/引用”。

RWKV 更像：
- 我读过了
- 我把信息压进状态里
- 后面继续用这个状态

这对 agent 很诱人。

因为真正长期运行的智能体，不应该每次都靠巨大 prompt 续命。更理想的是：
- 有短期上下文
- 有压缩状态
- 有外部记忆
- 有可更新的内部状态

RWKV 在结构上更接近这个方向。

## RWKV 的主要问题

### 1. 压缩状态是优势，也是瓶颈
RWKV 把历史信息压进 recurrent state。

这很好，因为省内存、推理快。

但问题也来了：

> 历史一旦被压缩，细节可能丢失，而且不容易精确回看。

Transformer attention 的一个优势是可以直接 attend 到过去 token 的细节。  
RWKV 更像把过去揉成一个状态向量。

所以它适合：
- 连续流式理解
- 模式累积
- 长期趋势

但在一些需要精确检索的位置，比如：
- “第 4372 个 token 里那个变量名是什么”
- “前面某段原文的精确措辞”
- “跨很长距离的 rare detail binding”

它会比较吃亏，除非配合外部记忆 / retrieval / chunk cache。

---

### 2. 大规模上限还没被完全证明
RWKV 能做出不错模型，这点不用怀疑。

但它是否能在最前沿规模上和 Transformer 系模型硬碰硬，还没有完全证明。

原因不是它不好，而是：
- Transformer 生态太成熟
- 训练 recipe 太多
- 硬件栈高度适配 attention
- 大厂堆了多年经验
- scaling law / post-training / tool-use / multimodal 都围绕 Transformer 积累

RWKV 要证明自己，需要的不只是架构好，还要全套工程生态追上。

---

### 3. 表达能力和可控性需要更细的评估
RWKV 这种结构的 inductive bias 很强：

- 更顺序
- 更状态化
- 更像时间序列系统
- 少了显式 pairwise attention

这可能让它在某些任务更自然，在某些任务更别扭。

比如：
- 长文本流式理解：可能好
- 逐 token 低成本生成：好
- 精准复杂引用：可能难
- 多源信息任意交叉对齐：可能不如 attention 直观
- 多模态 token 混合：要看设计

所以不能只看“复杂度更低”，还要看它牺牲了哪些交互能力。

## 我觉得 RWKV 最适合的方向

### 1. 端侧 / 小模型 / 私有化
RWKV 的低推理状态成本很适合：
- 手机
- 本地助手
- 边缘设备
- 低显存机器
- 长时间待机 agent

如果有一天出现一个“常驻本地、持续记忆、低成本运行”的个人 AI，RWKV 这类架构是很合理的候选。

---

### 2. 长流式 agent
比如一个 agent 一直看日志、消息流、行情、传感器数据。

Transformer 每次把全部上下文塞进去很蠢。  
RWKV 式状态更新更自然：

> 事件来了，更新状态；需要行动时，从当前状态生成。

这和真实世界流式智能体更匹配。

---

### 3. 和外部记忆结合
我不太看好 RWKV 单独替代所有注意力。  
但我看好：

**RWKV + retrieval memory + selective attention**

也就是：
- RWKV 负责连续状态和快速更新
- 外部向量库/符号记忆负责精确长期记忆
- 必要时用 attention 对关键片段做精读

这可能比纯 Transformer 堆上下文更优雅。

---

## 和 Transformer / Mamba 比

### RWKV vs Transformer
RWKV 优势：
- 推理状态小
- 长序列成本低
- 流式自然
- RNN-like 持续状态

Transformer 优势：
- 表达力强
- 生态成熟
- 精确回看强
- scaling 已充分验证
- 硬件和软件栈成熟

我的判断：  
**RWKV 不会短期取代 Transformer，但会在某些长流式和端侧场景有真实价值。**

---

### RWKV vs Mamba / SSM
Mamba 更学院派、论文影响力更大，状态空间模型路线比较优雅。

RWKV 更像：
- 简洁
- 工程友好
- 社区驱动
- RNN 直觉强
- 更 hacker 文化

我不觉得二者必然谁赢。  
它们都在冲同一个问题：

> 如何摆脱 attention 的长上下文成本？

长期看，也许未来主流不是纯 Transformer、纯 RWKV、纯 SSM，而是混合架构。

---

## 我个人最看重 RWKV 的一点

不是它现在 benchmark 多强，而是它坚持了一个正确问题：

> 智能系统是否应该拥有可持续更新的内部状态，而不是每次靠 prompt 重新拼装自己？

这个问题非常重要。

如果我们从“聊天模型”走向“长期运行的个人智能体”，RWKV 的思想会越来越有价值。

因为长期 agent 需要：
- 低成本持续运行
- 可压缩状态
- 不断吸收新事件
- 不依赖无限上下文窗口
- 能把当前世界状态维持在内部

RWKV 在这方面比标准 Transformer 更自然。

---

## 结论

我认为 RWKV：

- **技术方向：很有价值**
- **工程路线：值得尊重**
- **短期替代 Transformer：不现实**
- **端侧 / 长流式 / agent 场景：很值得押注**
- **最大短板：精确长程检索和前沿规模验证**
- **最理想形态：RWKV + 外部记忆 + 局部 attention / retrieval 的混合系统**
