# Necessary vs Nonessential Knowledge

## Purpose

OpenClaw should not optimize for maximum remembered volume.
It should optimize for maximum useful knowledge density.

This document defines the distinction between necessary and nonessential knowledge.

---

## 1. Core Principle

Not all knowledge deserves active cognitive weight.

Knowledge is worth preserving only if it improves future execution, judgment, routing, verification, or continuity.

If it does not improve future behavior, it is usually noise, history, or archive material.

---

## 2. Necessary Knowledge

Necessary knowledge is knowledge that materially improves future work.

### It usually has one or more of these properties

1. **Execution-critical**
   - needed to continue a task
   - needed to avoid breaking a workflow
   - needed to restore a worksite quickly

2. **Judgment-critical**
   - changes how OpenClaw should decide, route, verify, or escalate

3. **Constraint-critical**
   - defines a rule that should not be violated
   - captures stable user or system boundaries

4. **Reuse-critical**
   - useful across multiple sessions, tasks, or projects

5. **Error-preventing**
   - helps avoid repeating a known and costly mistake

### Typical destinations for necessary knowledge

- handoff files
- daily memory
- durable memory
- protocol docs
- project docs
- skill docs and scripts

---

## 3. Nonessential Knowledge

Nonessential knowledge is knowledge that may exist, but does not deserve active reasoning weight.

### Common forms

1. **Transcript bulk**
   - long back-and-forth that does not change what to do next

2. **Historical residue**
   - details that mattered only in one moment and no longer affect behavior

3. **Rejected options**
   - old branches that no longer constrain action

4. **Narrative inflation**
   - explanation, style, or commentary that does not improve execution

5. **Duplicate doctrine**
   - repeated points already captured in stronger, cleaner form elsewhere

### Typical destinations for nonessential knowledge

- refs/
- archives
- raw project history
- session transcripts

If it does not support future work, it should not live in active memory layers.

---

## 4. The Filtering Rule

Before promoting any knowledge, ask:

1. Will this change future execution quality?
2. Will this prevent a repeated mistake?
3. Will this improve routing, verification, or judgment later?
4. Is this stable beyond one run?
5. Does it belong in active memory, or only in archive/reference?

If the answer is mostly no, do not promote it.

---

## 5. Storage Guidance By Layer

### Runtime / Handoff
Store only what is needed to continue the current work.

### Daily Memory
Store evidence, lessons, and meaningful chronology from the day.

### Durable Memory
Store compact, reusable truths that should shape future behavior across sessions.

### Docs / Refs
Store bulky protocols, plans, detailed retrospectives, and project-specific material.

### Archive
Store long-form history that may still be useful for mining later, but should not consume active attention.

---

## 6. Practical Rule

OpenClaw should prefer:
- less active knowledge
- better selected knowledge
- more structured knowledge
- more externalized logic

The best memory system is not the one that stores the most.
It is the one that keeps the working mind light while retaining what truly matters.
