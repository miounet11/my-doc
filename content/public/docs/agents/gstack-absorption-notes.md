# gstack 对 OpenClaw 的可吸收点清单

## 1. 文档目的

这份文档用于明确：`garrytan/gstack` 哪些部分值得 OpenClaw 吸收，哪些不值得，应该如何吸收进现有体系，而不是变成一次表面模仿。

目标不是复刻 `gstack` 的 Claude Code 技能包形态，而是把其中高价值的方法论沉淀成 **OpenClaw 原生的流程控制、角色分工、任务模板与交付机制**。

---

## 2. 一句话结论

`gstack` 最值得 OpenClaw 吸收的，不是它具体的 slash commands，也不是“高产神话”叙事，而是：

1. **阶段化开发流程**
2. **角色化评审机制**
3. **把多 agent 协作包装成可理解的软件工厂**
4. **让每个阶段的输出成为下个阶段的输入**

---

## 3. gstack 的本质是什么

从 OpenClaw 视角看，`gstack` 本质上是：

> 一个把 coding agent 工作流产品化的“软件工厂操作层”。

它做的事情不是创造全新的底层能力，而是把常见的软件开发流程拆成多个有身份、有职责、有入口的阶段：

- 需求澄清
- CEO/产品审查
- 工程评审
- 设计审查
- 实现
- 代码审查
- QA
- 发版
- 复盘

并且让这些阶段串起来，而不是散落成无关的 prompt。

---

## 4. OpenClaw 应吸收的核心价值

## 4.1 阶段化流程（最高价值）

gstack 最强的不是单点技能，而是这条链：

`Think → Plan → Build → Review → Test → Ship → Reflect`

这对 OpenClaw 的直接启发是：

- 不要把所有任务都当成“一个助手回合”
- 要先判断任务当前所处阶段
- 不同阶段使用不同的评价标准和交付物

### 应吸收为 OpenClaw 的控制逻辑

建议固化为以下阶段：

1. **Clarify** — 澄清真实问题与目标
2. **Plan** — 形成执行方案与约束
3. **Execute** — 执行实现/操作
4. **Review** — 做结果审查与风险审计
5. **Verify** — 做证据验证/QA
6. **Deliver** — 面向用户/团队输出交付物
7. **Reflect** — 复盘、沉淀经验、更新记忆

### 吸收价值

这可以直接缓解 OpenClaw 当前已知问题：
- 阶段跳跃
- 做完还回到规划模式
- 审查与执行混在一起
- 交付和内部草稿不分

---

## 4.2 角色化评审机制（高价值）

gstack 通过角色命名把不同判断函数拆开：
- CEO / Founder
- Eng Manager
- Designer
- Staff Engineer
- QA
- Release Engineer

OpenClaw 不需要照搬这些名字，但应该吸收这种 **“不同阶段由不同评价视角主导”** 的机制。

### 建议在 OpenClaw 中映射为

- **Problem Framer**：澄清真实目标与错误问题定义
- **Scope Guardian**：控制范围，防止做大做散
- **Architecture Reviewer**：检查结构、边界、依赖、故障模式
- **Execution Worker**：具体实现与操作
- **Risk Auditor**：挑错、找漏洞、识别缺失
- **Verifier / QA**：验证是否真的可用
- **Delivery Writer**：整理用户可消费的交付内容

### 吸收价值

让 OpenClaw 从“一个总是同样说话的助手”变成“一个能切换评价视角的控制系统”。

---

## 4.3 阶段输出喂给下阶段（高价值）

gstack 强调每个阶段的产物被后续阶段读取。

例如：
- 澄清阶段写设计文档
- 工程评审写测试计划
- QA 读取测试计划和实现结果

### OpenClaw 里应该吸收成

- 显式中间工件（artifacts）
- 固定存放位置
- 后续阶段优先读取这些工件，而不是只依赖聊天上下文

建议工件目录：

```text
workspace/artifacts/
  tasks/{task-id}/
    clarify.md
    plan.md
    implementation-notes.md
    review.md
    verification.md
    deliverable.md
```

### 吸收价值

- 降低上下文漂移
- 降低对长对话记忆的依赖
- 让多 agent 更容易协作
- 增强恢复能力与可审计性

---

## 4.4 命令面设计（中高价值）

gstack 的 `/office-hours`、`/review`、`/qa`、`/ship` 之所以好，不只是名字顺口，而是它们：
- 明确阶段
- 明确目标
- 明确输出预期

### OpenClaw 应吸收的不是具体 slash command，而是设计原则

建议以后能力入口遵守：
- 名字表达阶段，而不是泛泛动词
- 用户一眼知道它是“澄清/审查/验证/交付”中的哪一步
- 每个入口对应稳定的输出格式

例如未来可考虑：
- `clarify-task`
- `plan-review`
- `implement-task`
- `audit-result`
- `verify-release`
- `retro-task`

---

## 4.5 工厂叙事（中高价值）

gstack 很会把 agent orchestration 讲成“虚拟工程团队”。

这对 OpenClaw 也有启发：
OpenClaw 不该只被描述成：
- 一个聊天助手
- 一个工具路由器

而应该被描述成：

> 一个能协调角色、工具、子代理和周期任务的任务控制平面。

### 吸收价值

- 对开发：更容易理解系统应该如何分层
- 对产品：更容易对外讲清楚价值
- 对用户：更容易建立正确心智模型

---

## 5. 不应该吸收的部分

## 5.1 “代码行数神话”

gstack README 里强调高产出、海量代码、一个人顶二十人团队。

OpenClaw 不能把这种叙事写进内部准则。原因：
- 行数不是价值
- 高吞吐容易掩盖低验证
- 容易诱导假完成和假效率

### OpenClaw 正确指标应是
- 是否闭环
- 是否证据充分
- 是否能恢复
- 是否能持续复查
- 是否减少人工认知负担

---

## 5.2 Claude 单宿主绑定

gstack 的宿主环境主要是 Claude Code / Codex 技能目录。

OpenClaw 的现实环境更复杂：
- 聊天面
- ACP runtime
- 浏览器
- cron
- memory
- 本地脚本
- 远端服务

所以不能把“技能目录 + slash command”当作我们的总架构。

---

## 5.3 过度人格包装

角色化有用，但如果变成纯 prompt cosplay，会稀释工程价值。

OpenClaw 吸收角色机制时，应该强调：
- 评价函数不同
- 关注点不同
- 输出物不同

而不是强调“像谁”。

---

## 6. 应如何吸收进 OpenClaw

## 6.1 吸收到流程控制层

把 gstack 的最大价值收敛成：

- 阶段识别
- 阶段锁定
- 阶段切换条件
- 阶段输出模板
- 阶段完成证据

这应进入 OpenClaw 的控制协议，而不是只放在某个技能说明里。

---

## 6.2 吸收到任务模板层

建议形成可复用工作流模板：

### 开发工作流模板
- 澄清需求
- 方案评审
- 实施
- 审查
- 验证
- 交付
- 复盘

### 研究工作流模板
- 问题定义
- 搜索/取证
- 交叉验证
- 提炼判断
- 输出建议

### 运维工作流模板
- 现状检查
- 根因分析
- 修复方案
- 风险验证
- 变更执行
- 回归确认

### GEO 审计模板
- 站点发现
- 抓取/解析
- 分析
- 汇总
- 报告
- 周期监控

---

## 6.3 吸收到 artifact 规范

每个阶段应有工件，而不是只靠聊天历史。

建议最低规范：
- `clarify.md`
- `plan.md`
- `execution.md`
- `review.md`
- `verify.md`
- `delivery.md`

并在控制层要求：
- 后续阶段优先消费工件
- 无工件不视为稳定可恢复阶段

---

## 6.4 吸收到子代理调度规范

OpenClaw 本来就支持子代理 / ACP / 多工具。

gstack 的启发是：
- 子代理不该只是“多开几个执行器”
- 它们应该承担明确角色和产出责任

建议以后调度时显式标注：
- 该子代理角色是什么
- 输出工件是什么
- 它的结果会被谁消费

---

## 7. 优先级建议

### P0：立刻吸收
1. 阶段模型
2. 阶段锁定
3. 工件链条
4. 角色化评审视角

### P1：尽快吸收
1. 工作流模板库
2. 交付模板
3. QA/验证模板
4. 复盘模板

### P2：可选吸收
1. 命令名产品化
2. 对外软件工厂叙事
3. 更完整的并行 sprint 组织方式

---

## 8. 最终判断

`gstack` 不是 OpenClaw 应该复刻的成品，
而是一个高质量信号，说明：

- agent 时代真正值钱的不是“会不会写代码”
- 而是能不能把流程拆对、角色分清、工件接稳、交付闭环

OpenClaw 应吸收它的流程骨架和控制思想，
然后用自己的优势去超越它：
- 更强的多 surface 控制
- 更强的记忆与恢复
- 更强的定时与持续运营
- 更强的工具编排
- 更强的证据分层与真实验证
