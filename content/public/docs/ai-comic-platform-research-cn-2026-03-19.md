# 中国市场 AI 小说改编漫画平台调研报告

日期：2026-03-19
范围：面向中国用户，聚焦“小说/故事 -> 长篇条漫/漫画”的自动化生产系统，目标支持 30–300 张、并发 10 本。
置信度：中高（基于公开 GitHub / Hugging Face / 项目说明，未对每个仓库做本地跑通验证）

---

## 1. 执行摘要

这不是一个“单模型出图”产品，而是一个 **内容工业化生产系统**。

如果目标是中国用户，并且希望支持：
- 从已有小说改编
- 长度 30–300 张
- 同时并发 10 本
- 用 LLM + 多线程批量生成

那么最合适的产品定位不是“AI 画漫画工具”，而是：

> **AI 小说改编条漫生产平台**

它的核心价值不在“生成一张好图”，而在：
1. 把长文本稳定改编为结构化分镜
2. 控制角色/服装/场景跨 100+ 张的一致性
3. 让用户可以单格返工，而不是整本重做
4. 支持多项目队列、成本控制、审核与导出

结论很明确：
- **LLM 负责改编、分镜、连续性控制、审校。**
- **图像模型负责 panel 级出图。**
- **任务调度系统负责并发、重试、缓存、成本和 GPU 利用。**
- **编辑工作台负责商业可用性。**

如果缺其中任何一个，最多只能做 demo，做不了生产平台。

## 2. 中国市场需求判断

### 2.1 目标用户画像

从需求强度和付费能力看，最值得优先服务的不是纯 C 端用户，而是以下三类：

#### A. 网文作者 / 小说工作室
需求：
- 把已有小说快速改成漫画或条漫
- 希望批量化、低成本、稳定输出
- 更在意改编速度和商业转化，不一定追求顶级画质

#### B. 漫画工作室 / 内容工厂 / MCN
需求：
- 同时跑很多项目
- 需要流程化、多人协作、审核和返工
- 需要“项目管理 + 队列 + 单格重绘 + 导出”能力

#### C. 短剧/漫剧团队
需求：
- 静态漫画只是中间产物，后续可能转视频、动态漫、配音
- 更重视“分镜脚本、镜头节奏、角色设定”的结构化资产

### 2.2 最适合切入的用户群

从你提出的目标看：
- 30–300 张
- 并发 10 本
- 小说改编

最匹配的是：

> **B2B / B2Pro：网文工作室、漫画内容工厂、短剧改编团队**

这决定了产品必须优先具备：
- 批处理
- 审核台
- 项目制管理
- 角色一致性
- 单格返工
- 成本统计

而不是优先做“简单易玩”的 C 端体验。

## 3. 中国用户最核心的真实需求

### 3.1 用户不是在买“AI 画图”
他们真正愿意买单的是：
- 小说自动改编成可生产的分镜脚本
- 一致性角色图和参考资产
- 批量生成 30–100+ 格
- 单格返工，而不是整本返工
- 可导出为团队可继续处理的工程资产

### 3.2 最痛的 5 个点
1. **角色不一致**：同一个主角 20 张之后就不像了
2. **长篇难控**：100 张以后剧情、服装、场景都容易飘
3. **改编费脑**：小说变漫画不是简单摘要，而是重写节奏与镜头
4. **返工成本高**：改一格牵一串
5. **多项目调度混乱**：并发项目越多，失败重试和成本越难控

### 3.3 推荐首刀切入点
最值得优先解决的不是“最高画质”，而是：

> **小说 -> 分镜脚本 -> 一致性角色图 -> 批量生成 -> 单格返工**

这是最接近付费价值的闭环。

## 4. 产品形态建议

### 4.1 不建议一开始做什么
不建议一开始追求：
- 整本小说全自动生成 300 张
- 全自动无人工介入
- 顶级电影级画质
- 复杂动作 / 战争 / 机甲 / 超大场面

这些会让成本、失败率、返工率一起爆炸。

### 4.2 最推荐的输出形态
优先中国用户时，建议主打：

#### 第一优先：竖屏彩色条漫 / 国漫风长条阅读
理由：
- 更符合中国移动端阅读习惯
- 更适合 AI 分段批量生成
- 更适合网文改编和爽点节奏
- 更容易转成短视频/动态漫素材

#### 第二优先：漫剧分镜稿
适合短剧/漫剧团队后续视频化。

### 4.3 推荐题材优先级
适合先做：
- 女频言情
- 豪门/霸总/逆袭
- 都市爽文
- 校园恋爱
- 古风言情
- 玄幻爽文
- 轻悬疑

不建议一开始做：
- 机甲
- 重战争
- 极强动作长篇
- 写实历史考据类

这些对一致性和作画控制要求更高。

## 5. 产品需要定义的关键维度

### 5.1 输入维度
系统必须明确支持哪些输入：
- 一句话故事点子
- 章节大纲
- 完整小说文本
- txt / docx / markdown / 粘贴文本
- 角色设定文档
- 参考图 / 角色参考照
- 世界观设定集

### 5.2 输出维度
系统输出应当至少支持：
- 分镜脚本 JSON
- 角色设定卡
- 场景设定卡
- 条漫 panel 图片
- 长图拼接 / PDF / 图片包导出
- 项目工程文件（便于后续继续编辑）

### 5.3 质量维度
必须建立一组可度量指标：

#### 叙事质量
- 剧情是否通顺
- 爽点/情绪点是否突出
- 是否保留原文核心桥段
- 对话是否自然

#### 视觉一致性
- 脸是否稳定
- 发型、服装、配色是否稳定
- 场景与道具是否连续

#### 分镜质量
- 景别是否合理
- 人物走位是否清楚
- 对白与画面是否匹配
- 单格信息密度是否合适

#### 工业效率
- 单格成本
- 单本成本
- 单本生成时长
- 返工率
- 成功率
- 并发稳定性

## 6. 推荐系统架构

你要做的是一条生产线，而不是“一个模型”。

### 6.1 总体架构

```text
小说输入
  ↓
内容理解层（解析 / 摘要 / 人物抽取 / 高光桥段抽取）
  ↓
改编层（节奏重写 / 条漫化改编 / 受众风格化）
  ↓
分镜层（结构化 panel JSON）
  ↓
角色与场景资产层（角色卡 / 服装卡 / 场景卡 / 风格卡）
  ↓
图像生成层（低清预览 -> 高清生产 -> 单格重绘）
  ↓
质检层（一致性 / 敏感内容 / 漏字漏图 / 排版检查）
  ↓
导出层（条漫长图 / PDF / 图片包 / 工程文件）
  ↓
任务调度层（并发、缓存、重试、监控、成本）
```

## 7. 核心模块拆解

### 7.1 小说理解引擎
职责：
- 章节切分
- 角色抽取
- 关系抽取
- 场景抽取
- 关键桥段抽取
- 情绪曲线分析

输出示例：

```json
{
  "chapter_id": "12",
  "summary": "女主进入总裁办公室并与男主爆发正面冲突",
  "characters": ["女主", "男主", "秘书"],
  "locations": ["总裁办公室"],
  "key_beats": [
    "女主闯入",
    "男主冷嘲",
    "情绪升级",
    "女主反击"
  ],
  "tone": "压迫感、强对抗",
  "target_panels": 28
}
```

### 7.2 改编引擎
职责：
- 忠实改编 / 节奏加速 / 爽点强化 / 恋爱线强化
- 将小说文本变成适合漫画阅读的桥段序列
- 控制 panel 数量和镜头节奏

输出：
- 漫画版大纲
- 每章 panel 预算
- 类型化风格参数（女频 / 男频 / 轻喜 / 悬疑）

### 7.3 分镜引擎
这是真正的核心。

每个 panel 必须结构化，而不是纯文案。

建议结构：

```json
{
  "panel_id": "12-07",
  "scene": "办公室夜晚",
  "characters": ["女主", "男主"],
  "shot_type": "中景",
  "camera_angle": "平视",
  "emotion": "压迫感",
  "action": "男主向前一步，女主下意识后退",
  "dialogue": ["你以为你能逃？"],
  "narration": "空气像绷紧的弦。",
  "continuity": {
    "location": "总裁办公室",
    "outfit": {
      "男主": "黑西装",
      "女主": "米色风衣"
    },
    "previous_panel_ref": "12-06"
  }
}
```

有了这种结构，后面才可能做：
- 缓存
- 批量重试
- 单格返工
- 角色一致性引用
- 成本回放与数据分析

### 7.4 角色一致性系统
这是一票否决模块。

需要管理：
- 主角色卡
- 配角卡
- 服装卡
- 发型/脸型/配色锚点
- 三视图 / 表情模板
- 参考图集
- 场景与道具资产

如果没有“角色资产系统”，长篇会迅速失控。

### 7.5 图像生成引擎
强烈建议采用双阶段：

#### 阶段 A：低成本预览
- 低清或草图
- 快速确认构图、人物、动作、情绪
- 便于批量筛错

#### 阶段 B：高清生产
- 最终高质量出图
- 支持局部修复、单格重绘
- 支持失败重试和优先级控制

这样比直接全高清生成便宜得多，也更适合大规模生产。

### 7.6 排版与导出引擎
负责：
- 条漫拼接
- 对话框与字体
- 阅读顺序校验
- PNG / JPG / PDF / ZIP 导出
- 后续动态漫可复用的时间轴资产导出

### 7.7 任务调度与并发系统
你提出“并发 10 本”，这意味着系统必须有工业调度能力。

建议任务粒度：
- project
- chapter
- segment
- panel
- redraw

而不是“一本书一个大任务”。

必须支持：
- 项目级队列
- panel 级子任务
- 优先级
- GPU 资源池
- 失败重试
- 任务幂等
- 缓存
- 成本和耗时统计

## 8. 并发 10 本的技术建议

### 8.1 并发不是简单多线程
“并发 10 本”真正意味着：
- 同时有多个项目在不同阶段流转
- 某些项目在跑分镜
- 某些项目在出角色图
- 某些项目在批量生 panel
- 某些项目在返工

所以你需要的是 **异步任务编排系统**，而不是单纯 Python 多线程。

### 8.2 推荐执行模型

#### 控制层
- API Server：接收项目、查询状态、配置模型、回传结果
- Orchestrator：负责把项目拆成 stage 和 task
- Queue：负责分发任务

#### Worker 层
- LLM Worker：小说理解 / 改编 / 分镜 / 审校
- Character Worker：角色卡生成与重绘
- Panel Worker：panel 出图
- QC Worker：质检、合规、连续性检查
- Export Worker：导出与归档

### 8.3 队列建议
每个 panel 单独建任务，并给出依赖关系：
- 只有角色资产准备完成后，panel worker 才开始
- 某些章节允许并行
- 某些 panel 必须串行（因为要依赖前后镜头状态）

### 8.4 核心调度策略
- 同一项目内：按章节和 panel 段并行
- 不同项目间：按优先级和资源预算调度
- 高清生成排在低清确认之后
- 返工任务优先级高于新图任务

## 9. 中国市场导向的业务建议

### 9.1 最适合的产品定位
建议定位：

> **AI 小说改编条漫生产平台**

比“AI 漫画创作工具”更准，因为你的用户真正要的是生产效率，不是创作玩具。

### 9.2 推荐首发能力（V1）
V1 只做最值钱的闭环：
1. 导入章节文本
2. 自动生成漫画分镜脚本
3. 生成角色设定并锁定一致性
4. 批量生成 30–80 格条漫
5. 支持单格返工与批量导出

### 9.3 V2 再做的能力
- 多项目并发队列
- 团队协作审核
- 视频/动态漫拓展
- 多模板（女频/男频/悬疑/古风）
- 私有部署和成本优化

### 9.4 商业模式建议
可考虑三条路：
- SaaS 按项目 / 按生成量收费
- 面向工作室的私有部署
- “软件 + 代运营/代生产”混合服务

对中国市场来说，**私有部署或半私有化方案** 可能更容易拿下工作室客户，因为他们会在意：
- 成本可控
- 数据可控
- API 稳定性
- 模型替换自由度

## 10. 关键风险

### 10.1 技术风险
- 长篇上下文太长导致改编漂移
- 角色一致性崩坏
- 图像模型在复杂动作和多人物场景失控
- 重试成本过高
- 10 本并发时 GPU 成本和失败率上升

### 10.2 产品风险
- 如果没有单格返工，用户无法接受
- 如果没有质量预览，高清成本会过高
- 如果不能沉淀结构化资产，只能做一次性出图

### 10.3 市场风险
- 中国平台审核和内容边界复杂
- 用户容易把“能出图”误以为“能商业化生产”，预期管理要清楚
- 如果定价按图走，容易陷入低毛利竞争

## 12. 对这些项目的综合判断

### 12.1 最适合参考的平台骨架
**AI-Comic-Generator**

原因：
- 最强调 JSON 结构化工作流
- 最贴近“故事 -> 角色 -> 分镜 -> 批量出图 -> 单格返工”的主链路
- 更适合做长篇漫画而不是视频

### 12.2 最适合参考的流程和协作形态
**openOii**

原因：
- 多 Agent 流程清晰
- 项目制与实时进度更完整
- 适合借鉴 orchestrator / review / regeneration 机制

### 12.3 最适合参考的底层图像能力方向
**DiffSensei**

原因：
- 研究层面最接近“定制漫画 panel 生成”
- 强调 controllable 和角色适配

### 12.4 不建议的做法
不要期望单独拿某个仓库直接变成你想要的平台。
更现实的做法是：
- 借 AI-Comic-Generator 的数据结构和编辑器思路
- 借 openOii 的多阶段 orchestration 思路
- 借 DiffSensei 的 panel 控制与角色适配方向
- 然后自己补上：中国市场、并发调度、成本控制、长篇返工、审核与导出能力

## 13. 推荐落地路线图

### 13.1 Phase 1：最小可用闭环（必须先做）
目标：证明“小说改编成可用条漫”是可行的。

新增：
- 多项目管理
- 批处理队列

- 审核台
- 成本统计
- 模型路由

### 13.3 Phase 3：规模化
目标：支持并发 10 本及更多。

新增：
- GPU 资源池
- 任务优先级调度
- 缓存与重试机制

- 私有部署 / 混合云部署
- 审核合规模块

## 14. 你现在必须先确认的 12 个产品问题

1. 主用户到底是工作室还是个人创作者？
2. 优先题材是什么？女频、男频、古风还是都市？
3. 输出是竖屏条漫还是页漫？
4. 输入按章节还是按整本处理？
5. 标准成品是 30 张、80 张还是 300 张？
6. 用户最优先要速度、成本，还是质量？
7. 用户是否接受先低清预览、后高清生产？
8. 是否必须做角色一致性锁定？（答案大概率是必须）
9. 是否必须支持单格返工？（答案也几乎是必须）
10. 并发 10 本是 10 本同时出图，还是 10 本全流程同时跑？
11. 你准备走公有云 API 还是自建/私有 GPU？
12. 你最终卖 SaaS、私有部署，还是代生产服务？

这些问题不明确，技术方案一定会反复推翻。

## 15. 我的最终建议

### 战略判断
这个方向有真实需求，而且中国市场存在明确场景：
- 网文改编
- 条漫生产
- 漫剧前置分镜
- 内容工厂批量制作

### 最优产品定位
> **中国市场 AI 小说改编条漫生产平台**

### 最优工程策略
> **结构化分镜 + 角色资产系统 + panel 级任务调度 + 单格返工工作台**

### 最重要的 4 件事
1. **结构化 panel JSON**
2. **角色一致性资产系统**
3. **单格返工能力**
4. **并发调度与成本控制**

如果只能先做好一件事，我会先做好：

> **小说 -> 分镜脚本 -> 角色一致性 -> 30–50 格可返工条漫闭环**

这比直接追求“300 张 + 并发 10 本 + 全自动”更有胜率。
